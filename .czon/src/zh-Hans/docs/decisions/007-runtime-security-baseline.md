# ADR-007: 运行时安全基线（Assume Breach）

## 状态

已接受

## 日期

2026-03-10

## 背景

TheWorld 的核心运行面建立在容器之上。容器不是虚拟机，和宿主机共享内核；业务工作负载一旦出现 RCE，风险会沿权限与挂载边界扩散。

我们不采用"容器天然安全"假设，而采用 Assume Breach 立场：

- 默认认为业务代码可能被攻陷
- 重点控制失陷后的横向扩散与宿主机影响
- 安全边界必须前置到架构与默认配置，而不是靠运维临时补救

## 决策

TheWorld 在运行时采用以下强制安全基线：

1. **默认非 root 运行**
   - 业务容器进程必须使用非 root 用户
   - 不因便利性（调试、权限省事）回退为 root 默认

2. **最小化 Linux capabilities**
   - 默认 `cap-drop=ALL`
   - 仅按需增补最小集合（例如 `NET_BIND_SERVICE`）
   - 禁止常态化授予高危 capability（如 `SYS_ADMIN`）

3. **禁止高危运行参数**
   - 禁止 `--privileged` 作为常态
   - 禁止默认启用 `hostPID`、`hostNetwork` 等强耦合宿主机模式

4. **禁止高危挂载与控制面暴露**
   - 禁止向业务容器挂载 `docker.sock`
   - 禁止挂载宿主机根目录 `/` 与敏感系统路径（如 `/etc`、`/proc`、`/var/lib/docker`）
   - 控制面（容器编排、Daemon 管理能力）不得直接暴露给工作负载面

5. **推荐启用用户命名空间防护**
   - 优先使用 `userns-remap` 或 rootless 运行模式
   - 目标是将容器内高权限身份映射为宿主机低权限身份，降低逃逸收益

6. **审计与归因为默认能力**
   - 关键操作必须可审计、可归因到 AI 身份和任务上下文
   - 受控输出通道（如 outbox 授权写入）属于安全边界的一部分

## 理由

### 1. 风险现实存在，且常由误配置放大

真实攻防中，"RCE + root + 高危挂载/特权参数"会把容器风险快速放大为宿主机风险。

### 2. 默认值决定真实安全上限

安全边界如果不是默认值，最终会在效率压力下被绕过。将安全约束写入默认配置，才能稳定执行。

### 3. 最小权限是成本最低、收益最高的控制手段

相比复杂防护体系，非 root 和最小 capability 能直接阻断大量高危利用链路。

## 后果

### 正面影响

✅ 显著降低"容器 RCE 直接升级为宿主机沦陷"的概率  
✅ 安全策略可审计、可自动化检查、可持续演进  
✅ 为多用户隔离和 Scheduler 演进提供统一安全边界

### 负面影响

⚠️ 初期会增加权限适配工作（目录 owner、端口绑定、启动脚本）  
⚠️ 某些旧脚本会因非 root/cap-drop 而失败，需要重构  
⚠️ 调试便利性下降（但属于有意约束）

## 例外机制

允许临时例外，但必须同时满足：

1. 有明确业务必要性与风险说明
2. 有例外记录编号和过期时间
3. 有替代方案与回退计划
4. 在审计记录中可追踪

未记录的例外视为违规配置。

## 落地检查项

- 镜像与运行参数默认使用非 root 用户
- 运行参数默认 `cap-drop=ALL`
- CI/部署阶段阻断 `docker.sock`、`--privileged`、宿主机敏感路径挂载
- 安全相关例外单独建档并定期复核
