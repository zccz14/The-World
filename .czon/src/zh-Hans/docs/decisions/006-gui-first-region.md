# ADR-006: Region 默认采用 GUI-first 容器

## 状态

已接受（实施中）

## 日期

2026-03-09

## 背景

TheWorld 的长期目标是让 AI 在真实世界中持续工作，而不是只在理想化 API 环境中运行。

现实约束是：大量系统仍然是半封闭或非 API 优先的。

- 许多关键流程只能通过 Web UI 完成（登录、验证码、人机验证、复杂表单）
- 许多业务系统没有完整公开 API，或 API 能力与 UI 能力不对等
- 纯 headless 路径在这些场景会形成能力断层

如果默认坚持 No GUI，系统会在功能完备性上被外部世界的封闭性反向限制。TheWorld 无法强迫外部系统开放 API，因此应优先适配现实。

## 决策

Region 默认镜像采用 GUI-first 策略：

1. **每个 Region 默认具备可远程访问的桌面能力**（独立 GUI 会话）
2. **访问入口统一由 TheWorld 代理**（规划路径：`/gui/:region/*`），不以裸露 Docker 端口作为主要入口
3. **默认安全边界为仅本机访问**（`127.0.0.1`），跨机器访问需额外隧道或反向代理认证
4. **优先保证稳定与能力完备性**，资源开销作为可接受成本
5. **用户模型以技术风险最小化为准**：
   - 短期保持现有 `agent` 执行语义的兼容性
   - GUI 基座原生用户（如 `abc`）通过兼容层处理
   - 不为“命名一致性”牺牲稳定性

## 理由

### 1. 能力完备性优先于资源节省

GUI 不是体验增强，而是能力底座。默认有 GUI 可以覆盖不可预知的非 API 场景，避免功能边界被外部系统锁死。

### 2. 现实适配优先于理想假设

“所有系统都应有 API”是理想状态，不是当前事实。TheWorld 的默认运行时应尽量贴近真实用户系统能力。

### 3. 默认即立场

默认配置决定系统真实能力边界。采用 GUI-first 可以把“可用能力”前置，而不是在失败后再补救。

### 4. 与 TheWorld 目标一致

TheWorld 强调 AI 在“世界中”工作。真实世界大量交互发生在图形界面与浏览器中，GUI-first 更符合该定位。

## 后果

### 正面影响

✅ 覆盖反爬/验证码/登录会话等关键场景  
✅ 提升可视化调试与人工接管能力  
✅ 增强系统对未知任务的鲁棒性  
✅ 减少“因外部系统不开放”导致的任务失败

### 负面影响

⚠️ 单 Region 资源占用上升（CPU/内存/显存）  
⚠️ 运行与安全面更复杂（远程桌面入口、会话管理）  
⚠️ 与当前 `agent` 路径假设存在迁移成本

### 风险缓解

- 默认仅绑定 `127.0.0.1`
- 统一通过 TheWorld 代理入口，避免端口散落
- 维持 `agent` 语义兼容，采用渐进迁移
- 预留 headless 作为降级路径（故障恢复，不作为默认）

## 实施范围

### 架构/运行面

- Region 默认镜像从 headless 基座迁移到 GUI 基座
- 每个 Region 保持独立 GUI 会话
- TheWorld Server 新增 GUI 代理路由

### 文档/认知面

- 将 GUI-first 明确为默认策略
- headless 改为例外策略和降级方案

## 非目标

- 当前阶段不追求公网直连 GUI
- 当前阶段不追求为节省资源而回退默认 headless
- 当前阶段不强制立即重构全部用户模型（`agent`/`abc`）

## 与现状关系

当前 MVP（v0.1）实现仍以 headless Region 为主。本 ADR 定义了下一阶段默认策略，实施将以兼容优先、逐步迁移方式推进。
